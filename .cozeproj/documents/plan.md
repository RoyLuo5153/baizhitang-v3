# 百芝堂V3双轨驱动赋能系统 — 实施计划

## 概述
为百芝堂（糖尿病慢病管理公司）重建培训赋能系统，核心是**双轨驱动**（过程线+结果线交叉验证→四象限分类→精准赋能→闭环验证）。5角色（新人/带教/培训老师/管理者/老板）、7层架构（基础→追踪→学习→评估→诊断→赋能→决策），技术栈 Next.js + Supabase，SCRM数据V1手动录入、V2对接API。

## 技术方案

| 维度 | 选择 | 理由 |
|------|------|------|
| 框架 | Next.js 16 (App Router) | 全栈一体化，API Routes替代Flask |
| 数据库 | Supabase (PostgreSQL) | 云端托管、100人并发无压力、实时订阅 |
| UI组件 | shadcn/ui + Tailwind CSS 4 | 高效开发、风格可定制 |
| 认证 | Supabase Auth + 自定义角色权限 | 5角色42项细粒度权限 |
| AI集成 | V1规则驱动（阈值+模板+预置方案库），V2接入coze-coding-dev-sdk LLM | V1可解释零成本，V2智能归因 |
| 对象存储 | Supabase Storage | 录音文件、培训资料、聊天截图 |
| 部署 | 沙箱自带公网域名 | 零配置上线 |
| SCRM对接 | V1手动录入，V2 API同步 | 渐进式，降低风险 |

## 功能模块

### 1F 基础层
**职责**：登录认证、角色权限、用户管理、系统配置

用户表字段示例：
```
users: id, username, real_name, phone, department, stage(1/2/3), status(active/paused/graduated), created_at
roles: id, name(trainee/mentor/teacher/manager/boss), display_name
user_roles: user_id, role_id
permissions: id, code, name, role_id
```

### 2F 追踪层
**职责**：新人看板、成长档案、录音质检、微信质检、阶段分布

录音质检记录：
```
qc_records: id, trainee_id, reviewer_id, type(recording/wechat), audio_url, 
  score_business, score_service, score_communication, score_process,
  ai_score, ai_analysis, manual_score, status(pending/reviewed), created_at
```

微信质检（4节点19动作）：加V沟通、用药关怀、复诊邀约、续方确认

### 3F 学习层
**职责**：资料库、题库管理、课程学习、学习路径、阶段启用

题库字段：
```
questions: id, level_id(1-21), question_type(choice/multi/judge/essay), 
  content, options, answer, explanation, difficulty, category, is_active
learning_progress: id, trainee_id, level_id, status(locked/active/passed/failed),
  attempts, best_score, passed_at
```

阶段启用：阶段一（1-7关理论）→阶段二（8-14关+实战）→阶段三（15-21关+达标下组）

### 4F 评估层
**职责**：日常考核、闯关评分、阶段评估、逐项对标

**核心逻辑——逐项对标（非加权打分）**：
每个指标独立判断合格/良好/优秀三档，不合格即触发对应赋能方案。不做加权汇总。

过程线各环节独立对标：
```
闯关成绩：合格70分 / 良好85分 / 优秀95分 → 不合格→补充理论闯关
录音质检-业务专业：合格70 / 良好85 / 优秀95 → 不合格→业务话术训练
录音质检-服务规范：合格70 / 良好85 / 优秀95 → 不合格→服务规范训练
录音质检-沟通技巧：合格70 / 良好85 / 优秀95 → 不合格→沟通技巧训练
录音质检-流程执行：合格70 / 良好85 / 优秀95 → 不合格→流程执行训练
微信质检4节点：各节点独立评分，不合格→对应节点专项训练
日常考核：合格70 / 良好85 / 优秀95 → 不合格→日常行为纠正
```

结果线各环节独立对标：
```
加V率：合格90% / 良好95% / 优秀98% → 不合格→加微承接能力不足→加微话术训练
用药率：合格90% / 良好95% / 优秀98% → 不合格→首次沟通未强调用药→用药沟通培训
面诊率：合格85% / 良好92% / 优秀96% → 不合格→面诊邀约能力不足→面诊邀约训练
接诊率：合格80% / 良好88% / 优秀93% → 不合格→接诊流程不熟练→接诊演练
签收率：合格85% / 良好92% / 优秀96% → 不合格→签收跟进不到位→签收话术训练
挂号率：合格80% / 良好88% / 优秀93% → 不合格→挂号引导能力不足→挂号引导训练
```

**阈值可配置**：所有合格/良好/优秀标准在管理后台可调整，默认值如上

四象限分类（基于不合格项数量）：
```
A类：过程线全合格 + 结果线全合格 → 达标
B类：过程线合格 + 结果线有不合格 → 机制/流程问题
C类：过程线有不合格 + 结果线全合格 → 可能运气型，过程有隐患
D类：过程线有不合格 + 结果线有不合格 → 个人能力问题，需全面复训
```

阶段评估：
```
stage_evaluations: id, trainee_id, stage, process_items(jsonb各环节评分), 
  result_items(jsonb各环节评分), quadrant(A/B/C/D), 
  unqualified_items(jsonb不合格项列表), evaluator_id, decision(pass/retry/advance), created_at
```

### 5F 诊断层
**职责**：双轨四象限、归因分析、SCRM数据导入、漏斗趋势、异常预警

四象限分类规则（逐项对标）：
```
A类：过程线全合格 + 结果线全合格 → 达标
B类：过程线合格 + 结果线有不合格项 → 机制/流程问题
C类：过程线有不合格项 + 结果线全合格 → 运气型，过程有隐患
D类：过程线有不合格项 + 结果线有不合格项 → 个人能力问题
不合格项 → 对应诊断 → 对应赋能方案（预置方案库）
```

诊断逻辑：结果数据 → 对标合格线 → 识别不合格项 → 锁定问题根因 → 推送对应赋能方案
例：加V率85% < 90%合格线 → 诊断"加微承接能力不足" → 推送"加微话术专项训练"

业务漏斗数据：
```
business_funnel: id, trainee_id, period(周/月), 
  total_patients, add_wechat_count, add_wechat_rate,
  medication_count, medication_rate, registration_count, registration_rate,
  consultation_count, consultation_rate, signing_count, signing_rate,
  created_at
```

### 6F 赋能层
**职责**：一键赋能、瓶颈预警、下组审批、资源推送、闭环验证

赋能方案库（按不合格项触发）：
```
empowerment_plans: id, name, target_metric(加V率/用药率/面诊率/...),
  description, resources[], tasks[], duration_days
empowerment_records: id, trainee_id, plan_id, triggered_by(metric_fail/manual),
  unqualified_item, before_value, status(active/completed/failed), 
  after_value, verified, created_at
```

闭环验证：赋能前四象限位置 → 执行方案 → 赋能后四象限位置 → 对比验证

### 7F 决策层
**职责**：全局健康度、新人ROI、阶段分布、漏斗趋势

全局健康度 = Σ(各新人合格项数 / 各项总数) / 新人总数 × 100

## 关键决策确认（Q1-Q10）

| 问题 | 确认结果 |
|------|---------|
| Q1 技术路线 | 重写全栈（Next.js + Supabase） |
| Q2 数据迁移 | 通过现有API搬运（685题+159份资料+用户） |
| Q3 SCRM数据 | 手动导入（截图/表格/手动输入），预留自动对接接口 |
| Q4 四象限逻辑 | **逐项对标**：每个指标独立判断合格/良好/优秀，不合格→诊断→赋能方案。四象限按"不合格项数量"分类，非加权总分 |
| Q5 微信质检 | V1上传截图+AI识别检查 |
| Q6 阶段流转 | 闯关全过→实战；连续4周A类→达标；连续2周D类→自动复训 |
| Q7 阶段启用 | 灰掉+提示解锁条件，不隐藏（让新人看到前方目标） |
| Q8 AI边界 | V1全部规则驱动（阈值判断+模板匹配+预置方案库），V2再接LLM |
| Q9 交付分批 | 第一批1F+3F+5F+7F，第二批2F+4F+6F |
| Q10 设计风格 | 米白暖色调性内现代化升级 |
| 阈值配置 | 管理后台可配置合格/良好/优秀标准 |
| 过程线逻辑 | 与结果线完全对齐，逐项对标 |

## 实施步骤

1. **阶段一：原型设计** — 加载 design-canvas 技能，按角色设计5套核心页面原型（新人闯关+成长档案、带教看板+赋能、管理诊断+看板、老师题库+资料、老板概览），完成原型后提交用户验收确认
2. **项目初始化+1F基础层** — coze init nextjs，配置Shadcn主题，创建Supabase表结构(users/roles/permissions)，实现登录认证+5角色权限+用户管理 | `src/app/`, `src/lib/supabase/`, `src/middleware.ts`
3. **3F学习层+4F评估层** — 685题导入、21关闯关界面、学习路径、答题评分、逐项对标评分（非加权）、阶段启用机制 | `src/app/(trainee)/learning/`, `src/app/api/questions/`
4. **2F追踪层** — 成长档案、录音上传+AI质检评分、微信质检4节点19动作、业务漏斗手动录入 | `src/app/(mentor)/qc/`, `src/app/api/qc/`, `src/app/api/business/`
5. **5F诊断层+6F赋能层** — 双轨四象限引擎（逐项对标）、归因分析（不合格项→问题→赋能方案）、预警触发、一键赋能方案库、闭环验证对比 | `src/app/(manager)/diagnosis/`, `src/app/(manager)/empowerment/`
6. **7F决策层+数据看板** — 全局健康度仪表盘、新人ROI、阶段分布图、漏斗趋势图、老板极简视图 | `src/app/(boss)/overview/`, `src/app/(manager)/dashboard/`
7. **数据迁移+集成测试** — 从现有系统迁移685题+159份资料+用户数据，全角色全流程冒烟测试，修复问题 | `scripts/migrate/`
8. **验收与交付** — 日志健康检查、完整接口冒烟测试、最终交付 | `AGENTS.md`

# 百芝堂V3双轨驱动赋能系统

## 项目概览

百芝堂（糖尿病慢病管理公司）培训管理系统，替代原有Flask+SQLite系统。核心实现"逐项对标"双轨诊断：每个指标独立判断合格/良好/优秀，不合格项自动触发赋能方案。四象限分类（按不合格项数量分类，非加权总分）。

### 版本技术栈

- **Framework**: Next.js 16 (App Router)
- **Core**: React 19
- **Language**: TypeScript 5
- **UI 组件**: shadcn/ui (基于 Radix UI)
- **Styling**: Tailwind CSS 4
- **Database**: Supabase (PostgreSQL) via coze-coding-dev-sdk
- **Auth**: Base64 token (V1, 后续升级JWT)

### 核心业务逻辑

- **双轨驱动**: 过程线（闯关+质检4维度+日常考核）× 结果线（加V率/面诊率/接诊率/签收率/用药率/挂号率）
- **逐项对标**: 每指标独立对标合格/良好/优秀阈值，非加权总分
- **四象限**: A(全合格)/B(结果不合格)/C(过程不合格)/D(全不合格)
- **三阶段**: 阶段一→二(闯关7关全通过)→三(连续4周A类)，连续2周D类触发复训
- **5角色**: trainee/mentor/teacher/training_manager/boss
- **系统盯人**: 角色动作分离 + 联动触发（演练提交→自动质检→低分赋能→通知导师）
- **角色权限**: 闯关解锁仅对trainee生效，其他角色全量可见可操作
- **首页分化**: trainee=成长计划 / training_manager=全局驾驶舱 / mentor=带教看板 / teacher=教务工作台 / boss=经营概览
- **新人看板vs档案**: 看板=预警盯人大屏(实时异常/逾期/低分/待审核) / 档案=业务追踪表(入职/培训周期/下组/月度指标)
- **预警联动**: 闯关逾期→预警+通知导师 / 演练低分→自动赋能 / 诊断D类→复训触发 / 即将下组→确认提醒
- **质检自动触发**: 演练提交→自动创建质检任务(非手动新建)，source_type追踪触发来源
- **四阶段成长模型**: 学习期(1周7天排课) → 练习期(2-4周19动作演练) → 独立期(2-3月) → 熟练期(3月+)
- **19核心动作4节点**: 首通电话(6动作30%) + 第三天回访(4动作25%) + 第五天预约(6动作30%) + 面诊当天(3动作15%)
- **5级评分标准**: 5分(全面完成有亮点) / 4分(按要求完成) / 3分(勉强完成) / 2分(明显遗漏) / 0分(未执行)
- **信任度公式**: 患者信任度 = 认知水平 × 专业感知 × 安全感
- **4类特殊情况**: 未按时用药/延迟用药/用药中断/不规律用药(10个补充动作)

## 根因修复原则（强制）

### 核心规则
每次修复必须从根本原因入手，禁止打补丁。

### 判断标准
| 做法 | 判定 | 示例 |
|------|------|------|
| 把错误架构改成正确的 | ✅ 根因修复 | router.push('/') → window.location.href='/' |
| 在错误架构上加防御代码 | ❌ 补丁 | setTimeout兜底、try-catch吞异常 |
| 消除问题产生的条件 | ✅ 根因修复 | 修复软导航→硬导航，cookie传播时序问题不存在了 |
| 绕过问题让用户看不到 | ❌ 补丁 | fallback默认值、Cache-Control no-cache |

### 强制5步流程（每步必须完成才能进入下一步）

**第1步：复现+定位断裂点**
- 必须先复现问题，确认问题真实存在
- 追踪完整链路，找到具体的断裂点（哪一行代码、哪个时序、哪个逻辑分支）
- 输出：断裂点位置 + 证据（代码行号/日志/请求响应）

**第2步：分析根因**
- 回答：为什么这个断裂点会存在？是设计缺陷、架构问题、还是逻辑错误？
- 如果答案是"某个值可能为空""某个时序可能不对"，继续追问为什么——直到追溯到架构/设计层面的原因
- 禁止停留在"加了检查就不会出错"这个层面

**第3步：设计方案（禁止补丁模式）**
- 方案必须回答：改了什么，让问题**不可能再发生**（不是"加了防护所以不太会发生"）
- 如果方案中出现以下关键词，大概率是补丁，需要重新思考：
  try-catch、setTimeout、fallback、默认值兜底、防御性检查、容错处理
- 例外：用户数据的输入校验属于正常业务逻辑，不算补丁

**第4步：验证**
- 说明修复后问题为什么不可能再出现
- 列出可能受影响的关联模块

**第5步：记录经验**
- 将本次修复的结构化记录追加到 `LESSONS.md`（症状/根因/修复/防错规则/类别/日期）
- 同类别经验累计 ≥3 条时，提炼预防规则写入 `RULES.md`

### 禁止的补丁模式
- try-catch 吞异常：错误被捕获但没修正原因
- setTimeout 等待：用时间换确定性，本质是猜
- fallback 默认值绕过失败：失败还在，只是不看
- 防御性 check 掩盖症状：检查存在但没解决为什么会出现异常值
- Cache-Control 等响应头绕过：没改流程，只是让浏览器不缓存

### 工单执行要求
- 每个修复项必须先完成5步流程再动手改代码
- 改完代码后在commit message中写明：根因是什么、为什么这样改能消除
- 如果发现工单描述的根因不对，停下来报告，不要在错误根因上执行

### 紧急hotfix例外通道

线上故障等不了5步走完时，允许先补丁止血，但必须遵守以下规则：

1. **标注**：补丁代码必须加 `// HOTFIX: 临时止血，根因待修` 注释
2. **时限**：24小时内必须完成根因修复并替换补丁代码，否则回滚
3. **追踪**：在项目TODO或工单中创建一条"根因修复"任务，状态标为"紧急待修"
4. **禁止**：不允许出现第二个 `// HOTFIX` 而第一个还没根修的情况——堆叠补丁 = 失控

#### 判断是否走hotfix通道

| 情况 | 走5步流程 | 走hotfix通道 |
|------|-----------|-------------|
| 用户无法登录/下单等核心链路阻断 | | ✅ 先止血 |
| 页面样式偏移、文案错误 | ✅ | |
| 间歇性报错但不阻塞主流程 | ✅ | |
| 数据一致性问题 | ✅（高风险，补丁容易掩盖） | |

## 结构性防错机制（强制）

**核心思维：规则靠人记住，结构让人不可能忘记。**

| 维度 | 规则型防错 | 结构性防错 |
|------|-----------|-----------|
| 靠什么 | 人记住规则 | 系统自动拦截 |
| 忘了会怎样 | 照样犯错 | 犯不了错（被结构挡住） |
| 执行成本 | 每次都要主动想起 | 不用想，结构替你做 |

### 6个结构性防错机制

#### 机制零：工单不落地 = 系统不可用

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 讨论完应该去执行 | 未关闭工单不推进，则拒绝新功能讨论 |

- 每个工单必须有明确状态：`待执行` / `执行中` / `已验证` / `已关闭`
- 会话启动时首读未关闭工单，第一个动作必须是"推进最优先的未关闭工单"
- 工单从`待执行`→`执行中`才能开始新功能讨论，否则拒绝
- 讨论了但不执行 = 讨论无效 = 系统不进步
- 类比：代码写了不提交 = 没写。工单写了不执行 = 没写

#### 机制一：角色确认 → 入口校验

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 讨论前发清单请大家确认 | 不确认就无法进入讨论 |

- 工单必须有角色分配（谁负责业务/谁负责代码/谁负责测试/谁负责审查）
- 角色分配是工单必填项，不是可选附加
- 没有角色分配的工单 = 无效工单，不能派发
- 类比：没有权限校验的路由 = 无效路由

#### 机制二：变更同步 → 变更锁

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 变更后@所有人通知 | 变更必须全员确认才能生效 |

- 角色变更记录写入 AGENTS.md 更新记录
- 每次派工时声明当前角色版本（如 v3.2）
- 执行者行为与角色版本不一致 → 任何人可标记"角色版本不匹配"
- 版本不一致 = 工单无效

#### 机制三：讨论纠偏 → 越界自动检测

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 任何人可以喊停 | 内嵌越界检测逻辑，每轮必跑 |

- 每轮讨论总结前3秒自检：各角色是否在自己职责内发言
- 自检清单：业务角色是否在评代码？代码角色是否在做业务判断？调度角色是否在写代码？
- 类比：pre-commit钩子——提交前自动跑，不靠人记得

#### 机制四：落地检查 → 无结论不流转

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 5分钟内输出结论 | 没有结论清单，任务无法进入下一步 |

- 结论清单3个必填项：① 结论（是什么）② 行动项（谁负责+什么时候）③ 文件更新（哪些文件需同步修改）
- 缺任何一项 = 结论不完整 = 不流转
- 类比：守门员不签字 = 不部署

#### 机制五：定期回顾 → 日程化自动触发

| 规则版（靠人） | 结构版（靠系统） |
|---------------|-----------------|
| 每周五记得回顾 | 日程自动触发，不回顾不开工 |

- 回顾是开始新工单批次的前置条件
- 回顾模板：① 本周有没有角色越界？② 哪个环节跑偏最严重？③ 角色版本需要更新吗？④ 防跑偏机制需要迭代吗？
- 类比：不通过测试就不能部署

### 交付前强制清单（Pre-Done Checklist）

每次调用 `done` 工具之前，**必须逐项过以下清单**，全部通过才能交付：

```
□ 未关闭工单是否已推进？是否有工单停留在"待执行"超过1个会话？（机制零）
□ 代码变更是否已 git commit + push？
□ 是否有bug修复？如有，是否走了5步根因流程？
□ 是否有 HOTFIX 代码？如有，是否创建了根因追踪任务？
□ AGENTS.md 是否需要更新（新功能/架构变化）？
□ DESIGN.md 是否需要更新（UI/样式/交互变化）？
□ 是否通过 test_run 验证？
□ 是否有遗留的 TODO/FIXME/HOTFIX？
□ 开工前是否查了 LESSONS.md / RULES.md 的相关经验？
```

### 经验闭环流程

```
遇到bug
  ↓
查LESSONS.md → 有记录 → 复用方案 → 修复 → 补充新信息
  ↓ 无记录
走5步流程（含记录到LESSONS.md）
  ↓
同类经验 ≥ 3条 → 提炼为预防规则写入 RULES.md
  ↓
下次编码时 RULES.md 自动约束 → 同类bug不再发生
```

### 经验自动回流（结构性卡点）

经验沉淀不是建议，是卡点——不沉淀就无法继续：

| 工作动作 | 结构性触发 | 不沉淀的后果 |
|----------|-----------|-------------|
| 修完bug | commit时必填根因+修复方式 | commit被拒 |
| 新功能上线 | 验收通过时必填业务笔记+体验坑 | 验收不签收 |
| 验收驳回 | 驳回时必填驳回原因+改进方向 | 驳回操作被阻止 |
| Hotfix止血 | 24h内根因跟进工单必附经验沉淀 | 工单无法关闭 |

### 经验升级机制

```
个案经验（1次）
  ↓ 同类出现 ≥ 3次
Pattern经验（N次）
  ↓ 评估：加规则还是改结构？
规则级 → 写入 RULES.md / 审核准则 / 检查清单
结构级 → 改代码结构/门禁规则/工作流卡点 → 同步写入 AGENTS.md 宪法级规则
```

- 结构级升级标准：同类经验 ≥ 3次 + 根因分析确认结构性缺陷
- 经验降级：连续6个月未再发生 → 降为"归档"状态，不参与开工前推送（但可检索）

### 开工前必查（强制）

接收工单/开始开发前，必须：

1. **首读未关闭工单**：检查是否有`待执行`/`执行中`的工单，有则先推进，禁止跳过未关闭工单开新功能（机制零）
2. 按工单涉及的模块查 LESSONS.md / RULES.md
3. 按工单涉及的错误类型查经验库
4. 将相关经验附在工单"历史教训"栏
5. 不附经验 = 工单不完整 = 不予开工

### 经验分类体系

| 类别 | 典型模式 |
|------|----------|
| 导航/路由 | 软导航cookie丢失、hydration错位、动态路由404 |
